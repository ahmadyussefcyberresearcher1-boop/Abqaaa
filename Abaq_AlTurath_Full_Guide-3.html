الخلفية | Background

Context and motivation for the project

العربية

مطعم عبق التراث (Heritage Fragrance) يمثل ثقافة وتراثاً فلسطينياً أصيلاً. المطاعم التراثية الفلسطينية تواجه ضغطاً متزايداً للتحول الرقمي. قبل هذا المشروع، كانت الحجوزات تُدار يدوياً، وتحديث القائمة يتطلب تدخلاً يدوياً، ولا يوجد نظام مركزي لتتبع بيانات العملاء. المشروع يعالج هذه الفجوة بحل رقمي شامل.

English

Abaq Al-Turath (Heritage Fragrance) represents authentic Palestinian culinary heritage. Heritage restaurants face increasing digital transformation pressure. Before this system, reservations were manual, menu updates required physical intervention, and no centralized customer tracking existed. This project bridges that gap with a comprehensive digital solution.

🍽️ الأطباق الفلسطينية المميزة في المشروع | Featured Palestinian Dishes

مسخن (Musakhan) — منسف (Mansaf) — مقلوبة (Makloubeh) — كبة نابلسية (Nabulsi Kibbeh) — حمص فتة — ورق عنب — مشويات — قطايف

المشكلة | Problem Statement

5 core problems identified in the report

📋

إدارة حجوزات يدوية Manual Reservation Management

تكرار الحجز، سجلات ورقية معرضة للضياع، لا تحديث فوري، صعوبة إدارة أوقات الذروة

قائمة طعام ثابتة Static Menu Limitations

لا يمكن تحديث القائمة بدون تدخل تقني، لا تحكم في الأسعار والتوافر، صعوبة إبراز الأطباق الموسمية

👤

لا إدارة لبيانات العملاء No Customer Data

لا تاريخ للحجوزات، لا تتبع للتفضيلات، لا تحليلات لبيانات الأعمال

💰

مدفوعات نقدية فقط Cash-Only Payments

تضييق قاعدة العملاء، قيود مصرفية فلسطينية، لا بدائل رقمية متاحة

🌐

لا وجود رقمي No Digital Presence

غير مرئي للعملاء الذين يبحثون أونلاين، لا منصة رقمية قابلة للبحث

Why Literature Review?

أنظمة إدارة المطاعم 2.1 Restaurant Management Systems

Brown & Lee (2020): الأنظمة الحديثة تحسّن الكفاءة 23% وتقلل أخطاء الحجز 18%. OpenTable (1998) رائدة في الحجوزات أونلاين. Toast POS وSquare تُثبت تفوق السحابة على الحلول المحلية.

📅

أنظمة الحجز الأونلاين 2.2 Online Reservation Systems

Kim & Park (2018): 72% من جيل الألفية يفضل الحجز أونلاين. NRA (2022): المطاعم بحجز أونلاين تحقق 30% زيادة في الحجوزات. تقنية Database Transactions + Optimistic Locking.

💳

التجارة الإلكترونية عربياً 2.3 E-Commerce in MENA

Statista (2023): سوق التجارة الإلكترونية MENA سيبلغ $49 مليار بحلول 2025. القيود المصرفية الفلسطينية تجعل Bitcoin بديلاً عملياً — Nakamoto (2008).

🔍 الفجوة التي يملأها المشروع | Research Gap

الحلول التجارية الموجودة (OpenTable, Toast POS) صُممت للسياق الغربي ولا تدعم: العربية RTL، تصنيف المأكولات الفلسطينية التراثية، خيارات الدفع المحلية. هذا المشروع يعالج هذه الثغرة بحل مُصمَّم خصيصاً للبيئة الفلسطينية.

What is UML? — Unified Modeling Language

A standardized visual language for designing and documenting software systems. Used before and during development to model structure (static) and behavior (dynamic). The 5 diagrams used in this project cover different perspectives of the system.

①

مخطط حالات الاستخدام | Use Case Diagram (§4.3)

WHO does WHAT — Actors & their interactions with the system

الهدف

يُظهر مَن يتفاعل مع النظام (الممثلون) وماذا يستطيعون فعله. يحدد حدود النظام وعلاقات الفاعلين بحالات الاستخدام، بما فيها علاقات «include» الإلزامية و«extend» الاختيارية.

Purpose

Shows WHO interacts with the system (Actors) and WHAT they can do. Defines system boundary and actor-to-use-case relationships, including mandatory «include» and optional «extend» relationships.

مخطط الفئات | Class Diagram (§4.4)

الهدف

يُظهر البنية الثابتة للنظام — الكلاسات وخصائصها (-) ووظائفها (+) والعلاقات بينها. هو المخطط الأكثر ارتباطاً ببرمجة OOP.

📌 نقاط مهمة في مخطط الفئات

  • (-) Private attributes: لا يمكن الوصول إليها من خارج الكلاس مباشرةً
  • (+) Public methods: يمكن استدعاؤها من الخارج — تُشكّل الـ API الداخلية للكلاس
  • enum: قيم محددة مسبقاً — role: user/admin، status: pending/confirmed/cancelled/completed
  • Singleton (Database): نمط تصميمي يضمن نسخة واحدة فقط من اتصال قاعدة البيانات طوال عمر التطبيق
  • FK (Foreign Key): userId في Reservation يربطها بـ User — علاقة 1 إلى متعدد

ما الفرق بين «include» و«extend» في مخطط حالات الاستخدام؟

What is the difference between «include» and «extend»?

▾
«include» علاقة إلزامية — حالة الاستخدام الأساسية تستلزم دائماً تنفيذ الحالة المُضمَّنة بدون استثناء. مثال في مشروعنا: "تسجيل الدخول" يتضمن دائماً "التحقق من بيانات الاعتماد". «extend» علاقة اختيارية — تُضاف فقط في ظروف أو شروط محددة. مثال: "إنشاء حجز" قد يمتد ليشمل "توليد عنوان Bitcoin" — فقط إذا اختار المستخدم Bitcoin كطريقة دفع، وإلا فلا.
«include»: Mandatory relationship — the base use case ALWAYS triggers the included one without exception. Example: "Login" always includes "Validate Credentials". «extend»: Optional relationship — only triggered under specific conditions. Example: "Create Reservation" may extend "Generate Bitcoin Address" — ONLY if the user selected Bitcoin payment, otherwise this path is skipped entirely.

ما هو نمط Singleton وكيف استخدمتموه في مخطط الفئات؟

What is the Singleton Design Pattern and how is it used?

▾
Singleton هو نمط تصميمي (Design Pattern) يضمن إنشاء نسخة واحدة فقط من كلاس معين طوال عمر التطبيق. استخدمناه في كلاس Database لضمان وجود اتصال واحد فقط بقاعدة البيانات MySQL في أي وقت. الكلاس يحفظ الاتصال في خاصية static $instance، وعند أي طلب جديد يُعيد نفس الاتصال المخزن بدلاً من إنشاء اتصال جديد. هذا يوفر الذاكرة، يتجنب تعارض الاتصالات، ويحسن أداء التطبيق بشكل ملحوظ.
Singleton is a Design Pattern that ensures only ONE instance of a class exists throughout the application lifecycle. We used it in the Database class to maintain a single MySQL PDO connection at any time. The class stores the connection in a static $instance property — on subsequent calls it returns the stored connection instead of creating a new one. This saves memory, avoids connection conflicts, and significantly improves application performance.
State Diagram

اشرح الحالات الأربع لدورة حياة الحجز وانتقالاتها

Explain the 4 states of the Reservation lifecycle and transitions

▾
دورة حياة الحجز لها أربع حالات: (1) Pending (معلق): الحالة الابتدائية — تُضبط تلقائياً عند إنشاء أي حجز. (2) Confirmed (مؤكد): المدير يوافق على الحجز — ينتقل من Pending. (3) Completed (مكتمل): بعد مرور تاريخ وقت الحجز بنجاح — ينتقل من Confirmed. (4) Cancelled (ملغي): الإلغاء من المستخدم (Pending→Cancelled) أو من المدير (Confirmed→Cancelled). Cancelled و Completed هما حالتان نهائيتان (Terminal) — لا مزيد من الانتقالات. يوجد أيضاً آلة حالات موازية لحالة الدفع: Pending → Paid أو Failed.
Reservation lifecycle has 4 states: (1) Pending: Initial state — automatically set when any reservation is created. (2) Confirmed: Admin approves the booking — transitions from Pending. (3) Completed: Reservation date/time has passed successfully — transitions from Confirmed. (4) Cancelled: Cancelled by user (Pending→Cancelled) or admin (Confirmed→Cancelled). Cancelled and Completed are both Terminal States — no further transitions possible. There's also a parallel payment state machine: Pending → Paid or Failed.

ما الفرق بين مخطط التسلسل ومخطط النشاط؟

What is the difference between Sequence and Activity diagrams?

▾
مخطط التسلسل (Sequence) يُركز على الرسائل والتواصل بين الكائنات عبر الزمن — يُظهر مَن يرسل رسالة لمَن وبأي ترتيب زمني. الزمن يتدفق من الأعلى للأسفل. استخدمناه لتوثيق كيف يتواصل العميل مع JavaScript ثم PHP API ثم MySQL. مخطط النشاط (Activity) يُركز على تدفق العمليات التجارية ونقاط القرار والأنشطة المتوازية — أشبه بمخطط الانسياب (Flowchart) لكن أكثر قوة. استخدمناه لتوثيق خطوات تسجيل المستخدم من بداية فتح الصفحة حتى نجاح التسجيل.
Sequence Diagram focuses on messages and interactions between objects over time — shows who sends what message to whom in what order. Time flows top-to-bottom. We used it to document how the Customer interacts with JavaScript → PHP API → MySQL. Activity Diagram focuses on business process flow, decision points, and parallel activities — similar to a Flowchart but more expressive. We used it to document the step-by-step user registration workflow from opening the page to successful registration.

Security

ما هو bcrypt ولماذا هو أفضل من MD5 وSHA؟

What is bcrypt and why is it better than MD5 or SHA for passwords?

▾
bcrypt هو خوارزمية تشفير مُصمَّمة خصيصاً لكلمات المرور. تفوقها على MD5/SHA: (1) Salting تلقائي: تُضيف بيانات عشوائية لكل كلمة مرور قبل التشفير — حتى لو مستخدمان لديهما نفس كلمة المرور، الـ hash يختلف تماماً. (2) Cost Factor قابل للتعديل: يمكن زيادة صعوبة الاختراق مع تطور قوة الحواسيب. (3) بطيئة بتصميم: تستغرق وقتاً أطول — هذا يجعل هجمات Brute Force وRainbow Tables غير عملية. MD5 وSHA سريعتان جداً ولا تدعمان Salting التلقائي — لا تُستخدمان أبداً لكلمات المرور في التطبيقات الحديثة.
bcrypt is a hashing algorithm specifically designed for passwords. Advantages over MD5/SHA: (1) Automatic Salting: adds random data before hashing — even if two users have the same password, their hashes are completely different. (2) Adjustable Cost Factor: difficulty can be increased as hardware improves. (3) Intentionally slow: takes longer to compute — making Brute Force and Rainbow Table attacks impractical. MD5 and SHA are too fast and lack automatic salting — NEVER use them for passwords in modern applications.
Database

ما هي SQL Injection وكيف منعتموها في مشروعكم؟

What is SQL Injection and how did you prevent it?

▾
SQL Injection هجوم يُدخل فيه المهاجم كود SQL ضار في حقول الإدخال. مثال: إدخال ' OR '1'='1' -- في حقل الإيميل قد يتجاوز المصادقة بالكامل. منعناه عبر PDO Prepared Statements: (1) نُجهّز الاستعلام مع placeholder (?) بدلاً من قيمة حقيقية. (2) MySQL يُحلّل الاستعلام ويخزنه. (3) نُمرر المعطيات منفصلةً عبر execute(). هكذا يُعامَل أي إدخال — مهما كان ضاراً — كبيانات وليس كأوامر SQL. قاعدة ذهبية: لا نسلسل بيانات المستخدم مع استعلامات SQL أبداً.
SQL Injection is an attack where malicious SQL code is inserted into input fields. Example: entering ' OR '1'='1' -- in the email field could completely bypass authentication. We prevented it via PDO Prepared Statements: (1) Prepare the query with a placeholder (?) instead of the actual value. (2) MySQL parses and stores the query structure. (3) Pass parameters separately via execute(). Any user input — no matter how malicious — is treated as DATA, never as SQL commands. Golden rule: NEVER concatenate user data directly into SQL queries.
Architecture

ما هي المعمارية ثلاثية الطبقات وما فوائدها؟

What is Three-Tier Architecture and what are its benefits?

▾
المعمارية ثلاثية الطبقات تقسم النظام إلى: (1) طبقة العرض (Presentation): HTML/CSS/JS — ما يراه المستخدم ويتفاعل معه. (2) طبقة التطبيق (Application): PHP RESTful API — منطق الأعمال والمعالجة. (3) طبقة البيانات (Data): MySQL — تخزين البيانات واسترجاعها. الفوائد: فصل المخاوف (Separation of Concerns) — يمكن تغيير قاعدة البيانات دون تغيير الواجهة. قابلية الصيانة والتوسع. سهولة الاختبار لكل طبقة بمعزل. إعادة استخدام الـ API مع تطبيقات مستقبلية (موبايل مثلاً).
Three-Tier Architecture divides the system into: (1) Presentation Layer: HTML/CSS/JS — what the user sees and interacts with. (2) Application Layer: PHP RESTful API — business logic and data processing. (3) Data Layer: MySQL — data storage and retrieval. Benefits: Separation of Concerns — you can change the database without touching the UI. Better maintainability and scalability. Each layer can be tested independently. The API can be reused by future applications (e.g., mobile app without changing the backend).
API

ما هو RESTful API ولماذا استخدمتموه؟

What is RESTful API and why did you use it?

▾
REST (Representational State Transfer) هو أسلوب معماري للتواصل بين الواجهة الأمامية والخلفية. مزاياه: (1) Stateless: كل طلب مكتفٍ بذاته — لا يعتمد على طلبات سابقة. (2) HTTP Verbs: GET (قراءة)، POST (إنشاء)، PUT (تحديث)، DELETE (حذف) — واضح ومنطقي. (3) JSON: تنسيق بيانات خفيف ومقروء يعمل مع أي تقنية. (4) فصل الواجهة عن الخادم: يمكن تطوير Frontend وBackend مستقلَّين. طبقناه في 13 endpoint تغطي جميع وظائف النظام.
REST (Representational State Transfer) is an architectural style for Frontend-Backend communication. Benefits: (1) Stateless: each request is self-contained — doesn't depend on previous requests. (2) HTTP Verbs: GET (read), POST (create), PUT (update), DELETE (remove) — clear and logical. (3) JSON: lightweight, human-readable data format that works with any technology. (4) Frontend/Backend separation: both can be developed independently. We implemented it across 13 endpoints covering all system functions.

لماذا اخترتم Bitcoin كوسيلة دفع إضافية؟

Why did you integrate Bitcoin as a payment method?

▾
السبب الرئيسي هو السياق الإقليمي الفلسطيني الفريد: القيود المصرفية تجعل الوصول لبوابات الدفع الدولية (Visa, PayPal, Stripe) صعباً أو مستحيلاً عملياً. Bitcoin كعملة رقمية لامركزية (Peer-to-Peer) لا تحتاج بنكاً وسيطاً ولا موافقة حكومية. وثّقنا هذا أكاديمياً في الفصل الثاني عبر Nakamoto (2008). من ناحية تقنية: نُولّد عنوان BTC فريد لكل حجز تلقائياً. هذا يوسع قاعدة العملاء ويعكس وعياً بالتحديات المالية الإقليمية الحقيقية.
The main reason is the unique Palestinian regional context: banking restrictions make international payment gateways (Visa, PayPal, Stripe) difficult or impossible in practice. Bitcoin as a decentralized peer-to-peer digital currency requires no banking intermediary or government approval. We documented this academically in Chapter 2 via Nakamoto (2008). Technically: we auto-generate a unique BTC address for each reservation. This expands the customer base and reflects genuine awareness of real regional financial challenges.

Methodology

ما هو نهج التطوير الذي استخدمتموه ولماذا؟

What development approach did you use and why?

▾
استخدمنا Agile-Inspired Incremental Development مستوحىً من Design Science Research (DSR) — Hevner et al. (2004). الطريقة التدريجية سمحت لنا بـ: (1) مراجعة مستمرة لكل مرحلة. (2) اكتشاف المشكلات مبكراً قبل التراكم. (3) تعديل المتطلبات بمرونة. المراحل: تحليل المتطلبات → تصميم النظام (UML) → تطوير الواجهة → تطوير الخادم → قاعدة البيانات → اختبار التكامل. لم نستخدم Waterfall الصارم لأن المتطلبات قد تتغير أثناء التطوير، وبدلاً من ذلك اعتمدنا على التكرار والتحسين.
We used Agile-Inspired Incremental Development based on Design Science Research (DSR) — Hevner et al. (2004). The iterative approach allowed: (1) Continuous review of each phase. (2) Early problem detection before they accumulate. (3) Flexible requirements adjustment. Phases: Requirements Analysis → System Design (UML) → Frontend Development → Backend Development → Database Implementation → Integration Testing. We avoided strict Waterfall because requirements may evolve during development — iterative refinement yields better results.
Database

لماذا اخترتم InnoDB وUTF8MB4 في قاعدة البيانات؟

Why did you choose InnoDB engine and UTF8MB4 charset?

▾
InnoDB: محرك قاعدة بيانات MySQL الذي يدعم: (1) ACID Compliance — Atomicity, Consistency, Isolation, Durability. (2) Foreign Key Constraints — يضمن سلامة العلاقة بين الجداول (user_id في reservations يجب أن يكون في users). (3) Transaction Support — عمليات متعددة كوحدة واحدة. UTF8MB4: ترميز نصي يدعم جميع أحرف العربية الكاملة + الرموز التعبيرية (Emoji) بدقة. UTF-8 العادية لا تدعم بعض أحرف Unicode الخاصة التي قد تظهر في الأسماء أو الوصف العربي. هذا ضروري لمشروع ثنائي اللغة.
InnoDB: MySQL storage engine that supports: (1) ACID Compliance — Atomicity, Consistency, Isolation, Durability. (2) Foreign Key Constraints — ensures referential integrity (user_id in reservations must exist in users). (3) Transaction Support — multiple operations as a single atomic unit. UTF8MB4: Character encoding that supports all Arabic characters + Emoji correctly. Regular UTF-8 doesn't support some special Unicode characters that may appear in Arabic names or descriptions. Essential for a bilingual system.
Scope

ما هي قيود المشروع الحالية وما العمل المستقبلي؟

What are the current limitations and future work?

▾
القيود الحالية: (1) نشر محلي فقط على Laragon — ليس على خادم إنتاج. (2) دفع Bitcoin محاكى — لا بوابة دفع حقيقية. (3) لا تطبيق موبايل. (4) لا إشعارات بريد إلكتروني حقيقية. (5) لا دعم متعدد الفروع. العمل المستقبلي: (1) تطبيق موبايل بـ React Native أو Flutter. (2) BTCPay Server لمعالجة Bitcoin الحقيقية. (3) محرك توصيات بالذكاء الاصطناعي. (4) نظام POS داخل المطعم. (5) لوحة تحليلات متقدمة. (6) معمارية متعددة الفروع.
Current Limitations: (1) Local deployment only on Laragon — not on production server. (2) Simulated Bitcoin payment — no real payment gateway. (3) No mobile application. (4) No real email notification system. (5) No multi-branch support. Future Work: (1) Mobile app with React Native or Flutter. (2) BTCPay Server for real Bitcoin processing. (3) AI-powered recommendation engine. (4) In-restaurant POS system. (5) Advanced analytics & BI dashboard. (6) Multi-branch architecture.
Literature

ما الفجوة في الأدبيات التي يسدّها مشروعكم؟

What research gap does your project fill?

▾
الأدبيات والحلول التجارية الموجودة (OpenTable, Toast POS, Square) صُممت للسياق الغربي وتفتقر إلى: (1) دعم العربية RTL — لا تدعم اللغة العربية بشكل أصلي. (2) تصنيف مناسب للمأكولات الفلسطينية التراثية — لا تفهم السياق الثقافي. (3) خيارات دفع محلية — تعتمد على بوابات دفع غير متاحة في فلسطين. مشروعنا يُصمَّم خصيصاً لهذه البيئة الثقافية والجغرافية. هذا ما يميزه عن مراجعة الأدبيات ويثبت الحاجة إليه.
Existing literature and commercial solutions (OpenTable, Toast POS, Square) are designed for Western contexts and lack: (1) Arabic RTL support — no native Arabic language interface. (2) Appropriate categorization for Palestinian heritage cuisine — no cultural context understanding. (3) Locally-relevant payment options — rely on gateways unavailable in Palestine. Our project is specifically designed for this cultural and geographic environment — this is what differentiates it from the literature and justifies its need.
Testing

كيف تحققتم من صحة النظام؟ ما أنواع الاختبار المستخدمة؟

How did you verify system correctness? What testing types were used?

▾
استخدمنا Manual Functional Testing عبر حالات اختبار إيجابية وسلبية. اختبرنا 4 وحدات رئيسية: المصادقة، القائمة، الحجز، الإدارة. الاختبارات الإيجابية: بيانات صحيحة تُنتج نتيجة متوقعة (تسجيل بنجاح، حجز نقدي/Bitcoin). الاختبارات السلبية: بيانات غير صالحة تُنتج رسالة خطأ مناسبة (كلمة مرور خاطئة، تاريخ ماضٍ، أكثر من 20 ضيف). أيضاً: اختبار أمان SQL Injection، اختبار التحكم بالوصول. النتيجة: 9/9 حالات = 100% نجاح.
We used Manual Functional Testing with positive and negative test cases. Tested 4 main modules: Authentication, Menu, Reservation, Admin. Positive tests: valid inputs produce expected results (successful registration, cash/Bitcoin reservation). Negative tests: invalid inputs produce appropriate error messages (wrong password, past date, 21+ guests). Also: SQL Injection security testing, Role-based access control testing. Result: 9/9 test cases = 100% success rate.
